Technology Architecture and Infrastructure

To support this transformation, OMSB will overhaul its technology architecture and infrastructure, moving from today’s fragmented setup to a modern, scalable, and secure digital platform. The target state architecture is defined by **modularity, interoperability, and use of cloud technologies**. Key aspects of the planned architecture include:

Cloud-First, Modular Architecture

OMSB will transition to a **cloud-native infrastructure**, leveraging the Oman Government Cloud and other approved cloud services for hosting all applications and data. The legacy monolithic systems will be refactored into modular **microservices** – each core function (registration, exams, CPD, accreditation, etc.) will be delivered as a set of containerized microservices with well-defined APIs. This microservices architecture, using containers (e.g. Docker) and orchestration (Kubernetes on cloud), ensures that each component can be developed, scaled, and updated independently, improving agility and reliability. For example, the exam management service can be scaled up around exam periods without affecting other services. Initially (Phase 1), a **hybrid cloud** approach may be used to migrate systems safely, but by Phase 2 OMSB plans a **multi-cloud deployment** (e.g. using both AWS GovCloud and Azure Government regions) with active-active failover for high availability. This multi-cloud strategy will enhance resilience by avoiding dependence on a single provider and enabling disaster recovery across platforms. **Infrastructure automation** (Infrastructure-as-Code tools like Terraform or Azure Resource Manager) will be adopted to standardize and speed up provisioning of resources. Overall, this approach brings scalability (systems can handle growth in users/data easily) and flexibility (new features or services can be plugged in with minimal impact), addressing OMSB’s current issues of rigidity and scalability.

API-First Integration Layer

A fundamental principle of the new architecture is **API-led connectivity**. OMSB will implement an **API gateway and integration middleware** that exposes all services via secure APIs and connects internal modules with external systems. In Phase 1, **RESTful APIs** will be developed for all major modules (with documentation via OpenAPI/Swagger) so that data can flow between OMSB’s systems and to authorized external consumers. By Phase 2, OMSB will introduce more advanced API capabilities like a **GraphQL API gateway** to allow flexible queries across multiple services in one call. An **event-driven integration mechanism** will be added as well - using an event streaming platform (e.g. Kafka), key events in one system (like “exam result published” or “user completed training”) will automatically trigger updates in other systems or notifications to users in real-time. This event-driven architecture will make OMSB services more responsive and decouple systems (so they sync via events rather than tight coupling). All APIs will use modern security (OAuth2/OpenID Connect for authentication, JSON Web Tokens, etc.) ensuring only authorized access. With this integration layer, OMSB can easily plug into external platforms as well – for instance, connecting with MOH’s systems or educational institutions via API calls rather than manual data exchanges. The **National Integration Platform (NIP)** will be used wherever possible to handle inter-agency data flows, in line with MTCIT guidelines. The end result is an **interoperable ecosystem** – both internally (all OMSB modules share data seamlessly) and externally (OMSB can exchange data with national and even international systems efficiently).

Centralized Data Platform

Data architecture is a crucial part of the strategy. OMSB will build a central **data lake and data warehouse** to break data silos and enable analytics. The **Data Lake** (using technologies like AWS S3 or Azure Data Lake Storage) will serve as a repository for raw, detailed data coming from all sources – LMS, MASAR e-portfolio, exam engine, HRIS, accreditation database, etc.. On top of this, a relational **Data Warehouse** (e.g. AWS Redshift or Azure Synapse) will be created to store integrated and cleaned data optimized for reporting and BI. By 2029, OMSB aims to have near **real-time data ingestion pipelines** (using streaming ETL tools like Kafka, AWS Glue or Azure Data Factory) feeding the warehouse, so dashboards reflect up-to-date information. Additionally, an **AI/ML Sandbox environment** (such as Databricks or SageMaker) will be provided for data scientists to develop predictive models using the collected data. A critical component of the data strategy is the implementation of the **Master Professional Index (MPI)** - this will be a unified database or identifier system that links all records related to a given individual (trainee, graduate, etc.) across all OMSB systems. The MPI will be implemented with appropriate master data management tools and likely integrated with national identifiers (e.g. Civil ID) to ensure consistency. An **HIE** (Health Information Exchange) middleware will also be introduced to support standards-based data exchange (using HL7/FHIR) between OMSB and clinical systems like the national EHR. By Phase 3, the data platform will even support advanced concepts like **federated learning** (collaborating on AI models with other entities without sharing raw data) to contribute to research or larger-scale AI initiatives. In short, OMSB’s data architecture will evolve into a unified, **analytics-ready platform** that turns data into a strategic asset.

Emerging Technology Enablement

The architecture is designed to accommodate emerging technologies that add value to OMSB’s mission. For example, OMSB will establish a **Blockchain-based Credential Ledger** as a supplementary component of the infrastructure. In Phase 1, a pilot using Hyperledger or similar will be conducted to issue a small number of digital certificates (credentials) on a blockchain for secure verification. By Phase 2, this will be scaled so that all OMSB certifications and possibly licenses are issued as **verifiable digital credentials** anchored on blockchain, complete with smart contracts to allow automated verification by employers or regulators. By Phase 3, OMSB plans to integrate its blockchain ledger with international networks – enabling a doctor’s credentials to be recognized across borders through a consortium of trust networks. Another component is the **AI & Automation Layer** - the architecture will include services for machine learning models and AI-driven processes. In Phase 1, OMSB will launch AI tools like an **“At-Risk Trainee Predictor”** that analyzes data across LMS, exams, etc. to flag individuals who might need support. Chatbots for common queries will also be deployed early. In Phase 2, more sophisticated AI features will come in (adaptive learning systems, AI-assisted grading, fraud detection in exam proctoring). By Phase 3, the architecture will support **autonomous workflows** – e.g. an AI system that can auto-approve license renewals under certain conditions or recommend actions to administrators (with oversight). The infrastructure will also support **tele-simulation and remote learning integration** - for instance, edge computing nodes (local processing units) will be set up at simulation centers to handle high-bandwidth VR/AR tasks with low latency, synchronizing key data back to the central cloud when needed. Essentially, the architecture is **future-ready** – built not just for current technologies but with the flexibility to incorporate new tools like Internet of Things (IoT) devices for training, or whatever next-gen tech emerges, without requiring a wholesale redesign.

Security and Identity Infrastructure

The new architecture will have **security built-in at every layer**. OMSB will implement a robust **Identity and Access Management (IAM)** system - all user authentication will be unified via **Single Sign-On**, likely using OAuth 2.0 / OpenID Connect standards federated with the national digital identity (e.g. TAM). **Multi-factor authentication (MFA)** will be enforced especially for privileged access and sensitive actions, adding an extra layer of security. The architecture will also move towards a **Zero Trust model** by Phase 2 – meaning that not only external but internal access to systems is continuously verified (implementing principles like device trust checks, just-in-time access, micro-segmentation of networks). A **Security Information and Event Management (SIEM)** system will be integrated to aggregate logs from all systems and detect threats in real-time. By Phase 3, OMSB will explore **AI-driven security tools** that use machine learning to detect anomalies or predict attacks (e.g. user behavior analytics for unusual activities). Data security will include **encryption at all levels** and possibly tokenization or anonymization techniques for personal data when used in analytics. Regular backups and a disaster recovery setup across the multi-cloud environment will ensure resilience. Additionally, the architecture will incorporate **privacy-by-design** - for instance, clear data segmentation between test and production, pseudonymization of personal data in non-prod environments, and capabilities to easily fulfill data subject requests (like a trainee requesting all their data be provided or deleted, if regulations require). All these measures ensure that while the architecture is open and integrated, it is also **secure and trustworthy**, protecting OMSB’s information assets.
